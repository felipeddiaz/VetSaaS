Plan de corrección de base de datos — VetCare SaaS

Ajustado para PostgreSQL · Sin datos reales en producción · Enfoque SaaS escalable

Contexto del problema

El modelo actual tiene una base correcta en términos de entidades y relaciones, pero carece de mecanismos estructurales fuertes que garanticen la integridad de los datos en un entorno multitenant. Actualmente, la coherencia entre organizaciones depende en gran medida de validaciones en la capa de aplicación, lo cual introduce riesgos importantes: un bug, una validación omitida o una operación directa sobre el ORM puede generar mezcla de datos entre clínicas.

Adicionalmente, existen riesgos en la gestión de datos históricos debido al uso de CASCADE en relaciones críticas, ausencia de constraints de negocio a nivel base de datos, y falta de una estrategia global de auditoría, aislamiento y eliminación lógica.

La meta de este plan no es solo corregir errores actuales, sino establecer una base sólida para un SaaS que pueda escalar sin comprometer integridad, trazabilidad ni mantenibilidad.

Paso 0 — Verificación inicial y baseline técnico

Por qué va primero: antes de introducir constraints, índices o cambios en relaciones, es necesario asegurar que el entorno de base de datos está correctamente configurado para soportar las capacidades avanzadas que se van a utilizar.

Qué verificar:

Confirmar uso de PostgreSQL (versión recomendada 13+).
Verificar que el encoding sea UTF8.
Confirmar timezone consistente (idealmente UTC en DB).
Revisar que no existan datos seed que violen futuras constraints.
Confirmar que el ORM está alineado con la DB (sin drift entre migrations y schema real).

Resultado esperado: entorno listo para aplicar constraints reales sin comportamientos inesperados.

Tiempo estimado: 1–2 horas.

Paso 1 — Corrección de comportamientos de borrado en datos históricos

Por qué es crítico: el uso de CASCADE en entidades clínicas o fiscales puede provocar pérdida irreversible de información. En un SaaS médico, esto no es aceptable ni técnica ni legalmente.

Qué cambiar y por qué:

Appointment.veterinarian → SET_NULL
La cita debe permanecer aunque el veterinario ya no esté activo.
MedicalRecord.veterinarian → SET_NULL
El historial clínico es permanente.
Prescription.veterinarian → SET_NULL
Una prescripción no puede desaparecer.
Invoice.owner → PROTECT
No se puede eliminar un propietario con facturas.
Invoice.pet → PROTECT
Las facturas deben permanecer incluso si la mascota se elimina del sistema lógico.
Invoice.appointment → PROTECT
No se puede eliminar una cita con impacto financiero.

Regla general:

Historia clínica → SET_NULL
Datos financieros → PROTECT
Detalles dependientes (ej. items) → CASCADE controlado

Tiempo estimado: medio día (sin datos reales simplifica todo).

Paso 2 — Tenant enforcement estructural (nuevo paso crítico)

Por qué es necesario: actualmente el aislamiento multitenant depende de filtros manuales. Esto no es suficiente en un SaaS. Se requiere un mecanismo estructural que reduzca al mínimo el riesgo de fuga de datos.

Estrategia en tres capas:

Capa 1 — OrganizationalModel consistente

Todos los modelos deben heredar de una base común con:

organization = ForeignKey(Organization)

Esto debe ser obligatorio en todas las entidades, incluyendo:

InvoiceItem
PrescriptionItem
MedicalRecordProduct

Eliminar modelos sin organization propio.

Capa 2 — Enforcement en ORM (automático)

Implementar:

Custom Manager / QuerySet que filtre automáticamente por organization.
Middleware o contexto que inyecte organization actual.

Ejemplo conceptual:

objects = TenantManager()

Esto reduce errores humanos en queries.

Capa 3 — Validación fuerte en modelo
Validaciones en clean()
Override de save() llamando full_clean()

Esto asegura que ninguna operación (API, admin, shell) pueda violar el tenant.

Capa 4 — Refuerzo en DB (PostgreSQL)

Cuando sea crítico (ej. facturación):

Constraints con SQL personalizado
Posible uso futuro de Row-Level Security (RLS)

Nota: Django no soporta bien constraints multitabla, pero PostgreSQL sí permite soluciones avanzadas si se requieren.

Tiempo estimado: 1.5–2 días.

Paso 3 — Constraints de coherencia multitenant entre FKs

Por qué sigue siendo necesario: aunque exista enforcement general, las relaciones específicas entre modelos deben validarse explícitamente.

Qué validar por modelo:

Pet → owner debe pertenecer a misma organization.
Appointment → pet y veterinarian deben coincidir en organization.
MedicalRecord → pet, veterinarian y appointment deben coincidir.
Invoice → owner, pet, appointment y medical_record deben ser coherentes.
InvoiceItem → referencias deben pertenecer a organization de la factura.

Dónde implementar:

Model.clean()
Serializer
Tests explícitos de cross-tenant

Adicional en PostgreSQL:

Se pueden agregar constraints SQL avanzadas donde aplique.

Tiempo estimado: 1–1.5 días.

Paso 4 — CheckConstraints de valores y rangos (DB-first)

Por qué en base de datos: PostgreSQL sí aplica estas constraints de forma estricta y eficiente. Esto elimina completamente estados inválidos.

Qué agregar:

En InvoiceItem:

quantity > 0
unit_price >= 0
subtotal >= 0

En Invoice:

tax_rate entre 0 y 1
total >= subtotal
subtotal >= 0

En Product/Presentation:

stock >= 0
price > 0

En Appointment:

end_time > start_time

En PrescriptionItem:

quantity > 0

Implementación:

Django CheckConstraint
o RunSQL si se requiere más control

Tiempo estimado: medio día.

Paso 5 — Regla XOR en InvoiceItem

Por qué es crítica: elimina ambigüedad en el tipo de línea y evita errores en inventario y facturación.

Regla:

exactamente uno de:
service
presentation

Implementación en PostgreSQL:

CHECK (
  (service_id IS NOT NULL)::int +
  (presentation_id IS NOT NULL)::int = 1
)

En Django:

CheckConstraint o RunSQL

Dependencia:

ejecutar después de migración de product → presentation

Tiempo estimado: 2–3 horas.

Paso 6 — Auditoría (nuevo)

Por qué es necesaria: en un SaaS necesitas trazabilidad completa.

Qué agregar en base model:

created_at
updated_at
created_by
updated_by

Opcional:

audit logs (tabla separada)

Esto permite:

debugging real
soporte a clientes
cumplimiento futuro

Tiempo estimado: medio día.

Paso 7 — Estrategia de eliminación lógica (soft delete)

Por qué: en SaaS casi nunca se elimina físicamente información crítica.

Qué implementar:

is_active = BooleanField(default=True)

O:

deleted_at = DateTimeField(null=True)

Aplicar a:

pets
owners
staff
productos

No aplicar a:

logs históricos
facturación cerrada

Esto evita pérdida de información y simplifica integridad.

Tiempo estimado: medio día.

Paso 8 — Índices compuestos

Por qué al final: dependen de cómo quedan los modelos después de las correcciones.

Qué agregar:

Appointment → (organization, date, status)
Invoice → (organization, status, created_at)
MedicalRecord → (organization, pet_id, date)
Pet → (organization, owner_id)
Staff → (organization, role, is_active)

Adicional importante:

(organization, id) implícito para joins eficientes

En PostgreSQL:

usar CREATE INDEX (no concurrente necesario aún)

Tiempo estimado: medio día.

Relación con arquitectura ORM + SQL

Este plan asume una estrategia híbrida:

ORM (Django) para:
modelado
validaciones de negocio
CRUD
SQL (PostgreSQL) para:
constraints críticas
optimización
enforcement estructural

Esto permite velocidad de desarrollo sin sacrificar integridad.
